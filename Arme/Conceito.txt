[#01 O QUE É MVP BENCHMARK — DEFINIÇÃO OPERACIONAL]
================================================================================

  MVP = Minimum Viable Proof of Performance
  Não é "mínimo produto" — é "mínima prova que funciona industrialmente"

  TRÊS CAMADAS DO MVP BENCHMARK:

  CAMADA 1 — FUNCIONAL (prova que o código faz o que deve):
    — Output determinístico: dado seed, resultado sempre igual
    — Checksum verificável: CRC32C do output == valor esperado
    — Tolerância de erro definida: F* = 23.158 ± 0.001
    — Custo: 1 execução, 0 hardware especializado

  CAMADA 2 — PERFORMANCE (medição industrial sem falso positivo):
    — Mediana de 31 amostras (ímpar: mediana exata, sem interpolação)
    — Aquecimento: 3 iterações descartadas (BTB, L1, decodificador)
    — Anti-DCE: force-use via asm volatile constraint
    — Anti-IRQ: detecta spikes > 10× mediana
    — Timer: cntvct_el0 (ARM64), rdtsc+lfence (x86), clock_gettime (ARM32)
    — Unidade: nanosegundos reais, não ciclos (Hz varia por DVFS)

  CAMADA 3 — INDUSTRIAL (equivalente ao que SPEC/CoreMark medem):
    — Throughput: bytes/segundo para operações de dados
    — Latência: nanosegundos por operação
    — IOPS: operações/segundo para I/O
    — Footprint: bytes de código + dados em uso real
    — Energia: joules/operação (quando medível)

  BENCHMARK MVP MÍNIMO PARA RAFAELIA:
    T1. fraf_iterate(Q16_ONE, 48)    → mediana, p5, p95
    T2. crc32c_buf(4KB)              → throughput GB/s
    T3. arena_alloc(64B)             → mediana latência
    T4. t7_step(1000×)              → mediana por passo
    T5. fsm_step(1000×)+Lyapunov    → mediana + classificação
    TOTAL: 5 kernels, ~155 medições, <1 segundo de execução

  POR QUÊ 31 AMOSTRAS E NÃO 100?
    — Mediana de 31: índice 15 (exato, sem média)
    — IQR: Q1=índice 7, Q3=índice 23 → robusto a outliers
    — 31 iterações de insertion sort: O(961) = trivial
    — 31 amostras × overhead de medição ~5ns = 155ns total overhead
    — 100 amostras adicionariam 69ns de overhead sem ganho estatístico

[#02 MODELO INDUSTRIAL — COMPARAÇÃO COM SPEC/COREMARK]
================================================================================

  COREMARK (EEMBC):
    — Mede: matriz, lista ligada, state machine, CRC
    — Problema: usa malloc, usa float, usa libc
    — Score: CoreMark/MHz (normalizado por frequência)
    — A78 @ 2.4GHz: ~7.5 CoreMark/MHz = 18.000 CoreMark total

  SPEC CPU2017:
    — Mede: workloads reais (compilador, física, IA)
    — Problema: inclui OS, libc, scheduler — não mede bare-metal
    — Não comparável com RAFAELIA

  RAFAELIA MVP vs CoreMark (A78 @ 2.4GHz, projeção):
    CoreMark:         18.000 pts    — com malloc, com libc
    RAFAELIA fraf:     0.81 ns/op   — sem libc, Q16, zero malloc
    RAFAELIA crc32c:  19 GB/s       — hardware instruction direto
    RAFAELIA arena:    2-3 ns/alloc  — 25× mais rápido que CoreMark malloc
    RAFAELIA fsm:      2 ns/trans    — lookup table, branch-free

  O PONTO CRÍTICO:
    CoreMark inclui fricção por definição (usa libc malloc).
    RAFAELIA ELIMINA a fricção — portanto não é comparável diretamente.
    A comparação correta é: "quanto de overhead CoreMark tem que
    RAFAELIA elimina" = diferença de 25×-60× por operação.

[#03 FRICÇÃO — TAXONOMIA COMPLETA E ESTRATÉGIAS DE REDUÇÃO]
================================================================================

  FRICÇÃO = energia desperdiçada entre intenção e silício

  TABELA DE FRICÇÕES POR CATEGORIA:

  ── FRICÇÃO DE ALOCAÇÃO ──────────────────────────────────────────────────
  malloc(64)      → 100-300ns  (lock+bins+mmap+metadata)
  jemalloc(64)    → 50-100ns   (thread-local cache, ainda tem overhead)
  tcmalloc(64)    → 20-60ns    (melhor mas ainda 10-20× mais lento)
  arena_alloc(64) → 2-3ns      (add + cmp = 2 instruções)
  alloca(64)      → 1ns        (adjust SP = 1 instrução, mas sem rollback)
  ZERO ALLOC      → 0ns        (variável na stack, sem chamada)

  ── FRICÇÃO DE MEDIÇÃO DE TEMPO ──────────────────────────────────────────
  gettimeofday()  → 500-1000ns (syscall completa)
  clock_gettime() → 100-300ns  (vDSO quando disponível)
  rdtsc (x86)     → 7-10 ciclos (lfence+rdtsc+lfence = ~4ns)
  cntvct_el0(A64) → 5 ciclos   (mrs = 1 instrução, ~2ns)
  PMCCNTR (A32)   → 3 ciclos   (mrc p15 = 1 instrução)
  OVERHEAD DE MEDIR A MEDIÇÃO: bench deve subtrair overhead do timer

  ── FRICÇÃO DE IMPRESSÃO ─────────────────────────────────────────────────
  printf("%d\n")  → 200-500ns  (format parse + va_args + FILE* lock)
  puts()          → 100-200ns  (strlen + write + buffer)
  write_str()     → 8-15ns     (strlen manual + syscall direto)
  raf_putu32()    → 5-10ns     (loop de divisão + write)

  ── FRICÇÃO DE GPIO (Arduino) ────────────────────────────────────────────
  digitalWrite(13,HIGH) → 3000-5000ns (50+ instruções, lookup, bounds)
  PORTB |= (1<<5)       → 125ns       (1 instrução SBI, 2 ciclos)
  PINB = (1<<5)         → 125ns       (toggle via PINB = 1 instrução)
  DIFERENÇA: 24-40×

  ── FRICÇÃO DE HASH ──────────────────────────────────────────────────────
  std::hash<string> C++ → 10-50ns/byte (overhead de objeto)
  strlen + loop XOR     → 0.5ns/byte  (1 instrução/byte)
  CRC32C hardware       → 0.05ns/byte (19 GB/s = ~0.05ns/byte no A78)
  SHA256 software       → 1.5ns/byte  (referência OpenSSL ARM64)

  ── FRICÇÃO DE BRANCH ────────────────────────────────────────────────────
  if/else (previsível)  → 0 ciclos extras (branch predictor correto)
  if/else (imprevisível)→ 14-20 ciclos (pipeline flush A78)
  CSEL/CMOV (branchless) → 1 ciclo     (seleção condicional sem flush)
  Lookup table           → 1-2 ciclos  (load + index)

  ESTRATÉGIA DE REDUÇÃO POR CAMADA:
    L1 (algorítmica):  escolher algoritmo com menos operações
    L2 (dados):        layout cache-friendly (sequential access)
    L3 (instrução):    branchless, SIMD, prefetch
    L4 (sistema):      eliminar syscall no hot path
    L5 (hardware):     usar instruções especializadas (crc32cx, fmla)

[#04 OS 39 TRICKSTOPATHCUTTER — CATÁLOGO COMPLETO]
================================================================================

  DEFINIÇÃO: "trickstopathcutter" = técnica que corta um caminho
  de overhead substituindo-o por um caminho direto ao hardware.

  ── GRUPO A: MEMORIA (1-8) ───────────────────────────────────────────────

  [T01] BUMP ALLOCATOR vs malloc
    CAMINHO CORTADO: lock+bins+fragmentation+metadata
    TÉCNICA: cursor += aligned_sz; return cursor_prev
    GANHO: 25-60× | CUSTO: sem free() individual
    APLICAÇÃO: qualquer struct de sessão fixa

  [T02] ARENA MARK/RESTORE vs malloc/free par
    CAMINHO CORTADO: rastreamento de ponteiros, free() individual
    TÉCNICA: mark=top; ... ; top=mark (restaura estado)
    GANHO: equivalente a free() em O(1) sem overhead
    APLICAÇÃO: funções recursivas, tentativas com rollback

  [T03] STACK ALLOC vs heap (alloca vs malloc)
    CAMINHO CORTADO: heap management, TLB pressure
    TÉCNICA: arrays de tamanho fixo declarados localmente
    GANHO: 10× | CUSTO: limite do tamanho da stack (~1-8MB)
    CUIDADO: VLAs são UB em C11 strict, usar arrays fixos

  [T04] ZERO COPY vs buffer copy
    CAMINHO CORTADO: memcpy de dados entre buffers
    TÉCNICA: passar ponteiro direto, não cópia do dado
    GANHO: elimina O(n) por operação | APLICAÇÃO: SPI/UART DMA

  [T05] BSS POOL vs dynamic pool
    CAMINHO CORTADO: mmap/malloc para pools de objetos
    TÉCNICA: static u8 pool[N][SZ] — em BSS, zero-init pelo linker
    GANHO: inicialização: zero tempo | CUSTO: tamanho fixo

  [T06] SLAB ALLOCATOR via BSS vs jemalloc
    CAMINHO CORTADO: size-class lookup em jemalloc
    TÉCNICA: pool de N slots de tamanho fixo + bitmap de uso
    GANHO: O(1) alloc/free sem fragmentação para tamanho único

  [T07] REGISTER PASSING vs stack passing
    CAMINHO CORTADO: store/load da stack para argumentos
    TÉCNICA: <= 8 args em ARM64 (x0-x7) ficam em registradores
    GANHO: elimina memory access | ABI: AAPCS64/SysV AMD64

  [T08] MEMSET via NEON vs loop byte
    CAMINHO CORTADO: 8 iterações por u64, 1 por byte
    TÉCNICA: stp xzr,xzr,[dst] — 16 bytes em 1 instrução
    GANHO: 16× para zeroing | ARM64: str q0 = 128 bits/instrução

  ── GRUPO B: COMPUTAÇÃO (9-16) ───────────────────────────────────────────

  [T09] Q16 FIXED-POINT vs float
    CAMINHO CORTADO: FPU pipeline, conversão de tipo, NaN handling
    TÉCNICA: s32 × s32 >> 16 = Q16 multiply em 1 instrução SMULL
    GANHO: 1-3× para operações simples | SEM libm, SEM -lm

  [T10] CLZ para LOG2 vs libm log2f()
    CAMINHO CORTADO: transcendental approximation (série Taylor)
    TÉCNICA: 31 - __builtin_clz(v|1) = log2 inteiro exato
    GANHO: 6× | CUIDADO: CLZ(0) = UB, usar (v|1)

  [T11] BRANCHLESS SELECT vs if/else
    CAMINHO CORTADO: branch prediction miss (14-20 ciclos)
    TÉCNICA ARM64: CSEL Rd, Rn, Rm, cond (1 ciclo)
    TÉCNICA C: mask=-(cond!=0); r=(a&mask)|(b&~mask)
    TÉCNICA x86: CMOV (conditional move)
    GANHO: 14-20 ciclos quando branch é imprevisível

  [T12] LOOKUP TABLE vs switch/if-chain
    CAMINHO CORTADO: comparações sequenciais O(n)
    TÉCNICA: array indexado por valor = O(1)
    GANHO: O(n)→O(1) | CUSTO: memória para tabela (cache pressure)

  [T13] LFSR vs rand() / random()
    CAMINHO CORTADO: lock global de rand(), heavy state
    TÉCNICA: s=(s>>1)^(-(s&1)&0xB400) = 3 instruções, período 65535
    GANHO: 20-50× | PROPRIEDADE: reproduzível, sem lock

  [T14] XOR SWAP vs swap com temp
    CAMINHO CORTADO: 1 instrução de store extra
    TÉCNICA: a^=b; b^=a; a^=b (3 XOR vs 3 mov)
    GANHO: elimina 1 instrução de memória
    CUIDADO: NUNCA usar com ponteiros iguais (a=b → resultado zero)

  [T15] BIT TRICK para ABS vs conditional
    CAMINHO CORTADO: branch para abs()
    TÉCNICA: mask=v>>31; (v+mask)^mask = abs sem branch
    GANHO: elimina branch imprevisível em 50% dos casos

  [T16] BITFIELD EXTRACTION via UBFX/BFI
    CAMINHO CORTADO: shift + mask manual (2 instruções)
    TÉCNICA ARM64: UBFX Rd, Rn, #lsb, #width (1 instrução)
    GANHO: 2→1 instrução | APLICAÇÃO: parsing de protocolos

  ── GRUPO C: PIPELINE E PARALELISMO (17-24) ──────────────────────────────

  [T17] DUAL ACCUMULATOR para FMLA
    CAMINHO CORTADO: cadeia de dependência de dados (serializa pipeline)
    TÉCNICA: v0 e v1 alternados em fmla, fadd final
    GANHO: throughput 2× | APLICAÇÃO: dot product, somatórios

  [T18] LOOP UNROLLING (×4, ×8)
    CAMINHO CORTADO: overhead de loop (branch + counter update)
    TÉCNICA: replicar corpo do loop N vezes, iterar N menos vezes
    GANHO: reduz overhead de 1/N | ×8 CRC32C = 64B/iteração
    LIMITE: aumenta código size, pode thrash I-cache

  [T19] PREFETCH MANUAL vs cache miss
    CAMINHO CORTADO: latência de cache miss (~200 ciclos DRAM)
    TÉCNICA ARM64: PRFM PLDL1KEEP,[ptr,#128] (2-3 iterações à frente)
    TÉCNICA x86: PREFETCHT0 [ptr+128]
    GANHO: esconde latência DRAM | CUSTO: pipeline slot de 1 ciclo

  [T20] INTERLEAVING de operações independentes
    CAMINHO CORTADO: dependência de dados que serializa OOO
    TÉCNICA: misturar instruções de unidades diferentes
    EXEMPLO: load(A) + compute(B) + store(C) em 1 ciclo OOO
    APLICAÇÃO: A78: 2×ALU + 2×FP + 1×Load + 1×Store por ciclo

  [T21] FUSED MULTIPLY-ADD (FMA/FMLA)
    CAMINHO CORTADO: multiplicação + adição = 2 operações
    TÉCNICA: FMLA Vd.2D, Vn.2D, Vm.D[0] = 1 operação
    GANHO: 2→1 instrução, sem arredondamento intermediário
    APLICAÇÃO: Q16_MUL via SMULL+SSHR em ARM64

  [T22] STP/LDP vs stores/loads separados
    CAMINHO CORTADO: 2 memórias → 1 instrução
    TÉCNICA: STP X1,X2,[X0] = store pair = 16 bytes em 1 instrução
    GANHO: 2→1 instrução | REQUER: alinhamento a 16 bytes

  [T23] STORE-TO-LOAD FORWARDING
    CAMINHO CORTADO: cache miss quando lê logo após escrever
    TÉCNICA: garantir que load acontece ≥4 ciclos após store
    GANHO: elimina pipeline stall | DIAGNÓSTICO: perf stat L1 misses

  [T24] SOFTWARE PIPELINING
    CAMINHO CORTADO: dependências entre iterações consecutivas
    TÉCNICA: separar load/compute/store de iterações distintas
    EXEMPLO: load[i+1] enquanto compute[i] enquanto store[i-1]
    GANHO: throughput ≈ pico teórico do hardware

  ── GRUPO D: SISTEMA E ABI (25-31) ───────────────────────────────────────

  [T25] CNTVCT_EL0 vs clock_gettime syscall
    CAMINHO CORTADO: EL0→EL1 + kernel + EL1→EL0 (~100-300ns)
    TÉCNICA: mrs x0, cntvct_el0 = leitura de registrador EL0
    GANHO: 20-60× | DISPONÍVEL: Linux ARM64 sem root

  [T26] RDTSC+LFENCE vs CLOCK_GETTIME
    CAMINHO CORTADO: syscall overhead no x86
    TÉCNICA: lfence; rdtsc (7-10 ciclos total)
    GANHO: 10-40× vs syscall | NOTA: vDSO reduz diferença

  [T27] SYSCALL DIRETO vs libc wrapper
    CAMINHO CORTADO: PLT + libc setup + errno + lock
    TÉCNICA: register x8=nr + svc #0 (ARM64)
    GANHO: 2-5× | CUSTO: código não portável por definição

  [T28] ZERO SYSCALL NO HOT PATH
    CAMINHO CORTADO: toda latência de syscall = 0 no loop principal
    TÉCNICA: todas as operações de timing via registrador de sistema
    GANHO: elimina latência de kernel do hot path completamente

  [T29] PIE SEM PLT (−fno-plt)
    CAMINHO CORTADO: PLT = indirection table = extra load + jump
    TÉCNICA: -fno-plt resolve chamadas diretamente pelo linker
    GANHO: 1 instrução por chamada | REQUER: linkagem estática ou RELRO

  [T30] SECTION SPLITTING (−ffunction-sections −fdata-sections)
    CAMINHO CORTADO: código morto no binário = I-cache pollution
    TÉCNICA: cada função em seção própria → linker descarta não-usadas
    GANHO: binário menor = melhor fit em I-cache

  [T31] LINK-TIME OPTIMIZATION (−flto=thin)
    CAMINHO CORTADO: otimizações perdidas entre translation units
    TÉCNICA: LTO analisa todo o programa no link, inline cross-TU
    GANHO: 5-20% de performance | CUSTO: tempo de build maior

  ── GRUPO E: TÉCNICAS INÉDITAS RAFAELIA (32-39) ──────────────────────────

  [T32] TOGGLE VIA PINx (Arduino)
    CAMINHO CORTADO: read-modify-write do PORTx (3 instruções)
    TÉCNICA: escrever 1 em PINx = toggle de PORTx em 1 instrução
    GANHO: 3→1 instrução | FREQUÊNCIA MÁXIMA: 4MHz toggle @ 16MHz

  [T33] CRC32C COMO HASH DE SESSÃO
    CAMINHO CORTADO: SHA256 (64 rounds) para integridade leve
    TÉCNICA: crc32cx de 8B em 1 ciclo → 19GB/s de "hash"
    LIMITAÇÃO: não é criptográfico, colisões em 2^32
    APLICAÇÃO: integridade de sessão local (HashVivo placeholder)

  [T34] WATCHDOG COMO OSCILADOR (AVR)
    CAMINHO CORTADO: RTC externo (DS3231 + I2C) para base de tempo
    TÉCNICA: WDT @ 1Hz = acorda de Power-Down sem hardware extra
    CUSTO: ~$0 em componentes | PRECISÃO: ±10% (RC interno AVR)

  [T35] ADC SIGMA-DELTA VIA COMPARADOR ANALÓGICO
    CAMINHO CORTADO: ADC externo de alta resolução (MCP3204, etc)
    TÉCNICA: 1-bit comparador + oversampling 4096× → 12 bits virtuais
    CUSTO: zero componentes extras | VELOCIDADE: ~1kSPS para 12 bits

  [T36] PHI64 COMO INDEX HASH (Knuth multiplicativo)
    CAMINHO CORTADO: modulo (divisão = 20-40 ciclos)
    TÉCNICA: (key × PHI64) >> (64-bits) = índice uniforme
    GANHO: divisão→multiplicação = 1 ciclo | DISPERSÃO: uniforme provada

  [T37] FIBONACI-RAFAEL COMO PRNG DETERMINÍSTICO
    CAMINHO CORTADO: rand() com lock global, estado compartilhado
    TÉCNICA: fraf_next(seed) = Q16 sequence com período definido
    PROPRIEDADE: ponto fixo F*=23.158 como valor de saída esperado
    APLICAÇÃO: teste de convergência = selftest automático

  [T38] ARENA MARK COMO CHECKPOINT DE ESTADO
    CAMINHO CORTADO: cópias profundas de estado para rollback
    TÉCNICA: raf_arena_mark() = salva cursor; raf_arena_restore() = reverte
    GANHO: O(1) checkpoint vs O(n) cópia de estado
    APLICAÇÃO: tentativas de parse, computação especulativa

  [T39] KERNEL MARKOV PLIMEX SEM SO
    CAMINHO CORTADO: scheduler do SO (CFS, preemption, lock)
    TÉCNICA: matriz de transição Q16 → probabilidade de recurso
    PROPRIEDADE: distribui CPU/RAM/IO sem system call
    EQUIVALÊNCIA: stochastic scheduler em 20 linhas de C

[#05 ASM NOSTUB NOMALLOC — ARQUITETURA COMPLETA]
================================================================================

  NOSTUB = sem _start da libc (crt0.o), sem __libc_start_main
  NOMALLOC = sem malloc, sem heap, sem brk(), sem mmap para dados

  POR QUE NOSTUB?
    libc _start faz antes do main():
    1. Inicializa errno (TLS allocation)
    2. Chama __register_atfork, __pthread_initialize_minimal
    3. Configura stdio (FILE* buffers = malloc interno)
    4. Chama _dl_aux_init (dynamic linker auxiliary)
    5. Chama __libc_csu_init (construtores C++)
    6. Chama main()
    7. Chama exit() com atexit handlers
    TOTAL: ~50µs de startup, 100KB de memória, zero controle

  NOSSO _START (ARM64, 7 instruções):
    _start:
        mov     x29, xzr       # frame pointer = 0
        mov     x30, xzr       # link register = 0
        and     sp, sp, #-16   # SP alinhado 16 bytes (AAPCS64)
        bl      raf_entry      # chama C entry
        mov     x8, #93        # SYS_exit
        mov     x0, xzr        # code=0
        svc     #0             # exit(0)

  ESTRUTURA DE MEMÓRIA NOSTUB/NOMALLOC:
    .text   → código (read-execute, nunca write)
    .rodata → constantes, strings (read-only)
    .data   → dados inicializados (raro: preferir .rodata ou stack)
    .bss    → arrays estáticos não inicializados (zero pelo kernel)
    stack   → variáveis locais, argumentos de função
    HEAP    → NÃO EXISTE
    ARENA   → no .bss, gerenciado pelo código

  ENDEREÇAMENTO ADEQUADO EM ARM64:
    PC-relative (ADRP + ADD):
      adrp x0, symbol          # x0 = page de symbol (±4GB)
      add  x0, x0, :lo12:symbol # x0 = endereço exato
      Uso: acessar .rodata, .bss de qualquer posição

    Base + offset imediato:
      ldr x1, [x0, #32]        # carrega offset fixo
      Uso: structs, campos de registro

    Base + registro × scale:
      ldr x1, [x0, x2, lsl #3] # carrega base[idx] onde cada elem=8B
      Uso: arrays de 64-bit

    Pre-indexed (atualiza base antes):
      ldr x1, [x0, #8]!        # x0 += 8, depois carrega
      Uso: iteração com atualização de ponteiro

    Post-indexed (atualiza base depois):
      ldr x1, [x0], #8         # carrega, depois x0 += 8
      Uso: loops de leitura sequencial

    Pair load/store:
      ldp x1, x2, [x0, #0]     # carrega 16 bytes em 2 regs
      stp x1, x2, [x0, #0]     # armazena 16 bytes
      Uso: qualquer acesso de 16 bytes alinhados

[#06 HOTSWAP IN-RUNTIME — MUTAÇÃO DE FLAGS E CÓDIGO]
================================================================================

  CONCEITO: mudar o comportamento do código EM EXECUÇÃO
  sem parar o processo, sem recompilar, sem reiniciar.

  NÍVEL 1 — FLAGS DE CAPACIDADE (mais simples, mais seguro):
    — u32 global com bits de feature
    — Setado na inicialização via CPUID/HWCAP
    — Consultado em cada chamada crítica
    — HOTSWAP: pode ser mudado por signal handler ou thread

    static volatile u32 RAF_CAPS = 0;
    #define RAF_CAP_CRC32C  (1u<<0)
    #define RAF_CAP_NEON    (1u<<1)
    #define RAF_CAP_SVE     (1u<<2)
    #define RAF_CAP_MOCK    (1u<<31)  /* modo teste/simulação */

    /* Hotswap: mudar CAP em runtime */
    void raf_caps_enable(u32 mask)  { RAF_CAPS |=  mask; }
    void raf_caps_disable(u32 mask) { RAF_CAPS &= ~mask; }

  NÍVEL 2 — TABELA DE FUNÇÕES (dispatch table):
    — Array de ponteiros de função indexado por capability
    — Trocar ponteiro = trocar implementação sem branch no hot path
    — HOTSWAP: atômico via CAS (compare-and-swap)

    typedef u32 (*crc_fn)(const u8*, usize);
    static volatile crc_fn G_CRC_FN = crc32c_software;

    void raf_hotswap_crc(bool8 use_hw) {
        /* Barreira ANTES de trocar */
        __asm__ volatile("dmb sy":::"memory");
        G_CRC_FN = use_hw ? crc32c_hardware : crc32c_software;
        /* Barreira DEPOIS: garante visibilidade */
        __asm__ volatile("dmb sy":::"memory");
    }

    /* Hot path: sem branch, sem if */
    u32 crc32c_dispatch(const u8* buf, usize len) {
        return G_CRC_FN(buf, len);  /* 1 load + 1 indirect call */
    }

  NÍVEL 3 — TRAMPOLINE PATCHING (código muda código):
    — Em runtime, sobrescreve os primeiros bytes de uma função
    — Com um salto incondicional para nova implementação
    — USADO EM: hotpatching de kernel Linux, DTrace probes
    — CUIDADO: requer páginas com permissão WRITE+EXEC
    — Em Android/Termux: mprotect() antes de escrever na .text

    void raf_patch_trampoline(void* target, void* replacement) {
        /* ARM64: B imm26 = 0x14000000 | (offset/4) */
        u32* code = (u32*)target;
        s64 offset = (char*)replacement - (char*)target;
        u32 branch  = 0x14000000u | ((u32)(offset/4) & 0x3FFFFFFu);
        /* Torna página writable */
        raf_mprotect(page_of(target), 4096, PROT_RWX);
        /* Barreira de instrução antes e depois */
        __asm__ volatile("isb":::"memory");
        *code = branch;
        /* Cache coherence: IC IVAU = invalidate instruction cache */
        __asm__ volatile("ic ivau, %0"::"r"(code):"memory");
        __asm__ volatile("dsb ish":::"memory");
        __asm__ volatile("isb":::"memory");
    }

[#07 MACHINE CODEX — REDUÇÃO DO CICLO DE MÁQUINA]
================================================================================

  MACHINE CODEX = conjunto de regras que reduzem ciclos de CPU
  ao mínimo físico possível para cada operação.

  REGRA MC-01: Uma instrução por loop iteration (não zero)
    — O hot path de qualquer loop deve ter pelo menos 1 instrução útil
    — Se o loop body é <4 instruções: considerar unrolling

  REGRA MC-02: Latência de instrução ≠ throughput
    — FMLA ARM64: latência=4c, throughput=1/2c
    — Para saturar: mínimo 4 instruções FP independentes no voo
    — Dual accumulator resolve automaticamente

  REGRA MC-03: Branch predict miss = 14-20 ciclos perdidos
    — Qualquer branch com <90% de previsibilidade = candidato a branchless
    — Medir: perf stat -e branch-misses ./programa

  REGRA MC-04: Cache miss L1 = 4c, L2 = 12c, L3 = 36c, DRAM = 200c
    — Todo dado crítico deve caber em L1 (64KB típico)
    — Structs: ordenar campos por frequência de acesso
    — Arrays: access pattern deve ser sequencial (stride-1)

  REGRA MC-05: Instrução de sistema (ISB, DSB, DMB) = 5-20c
    — Usar apenas quando necessário (fence antes de timer)
    — ISB: flush de pipeline (mais pesada)
    — DSB: drain de memória (média)
    — DMB: ordena acessos (mais leve)

  REGRA MC-06: SVC #0 = 100-300ns (1600-4800c @ 16MHz AVR = infinito)
    — Minimizar syscalls no hot path
    — Batch: acumular dados e fazer 1 write() em vez de N

  REGRA MC-07: Registradores de sistema (mrs/msr) = 3-5c
    — cntvct_el0, nzcv, pmccntr = acesso direto sem privilégio
    — Preferir a syscall de timing em qualquer hot path

  REGRA MC-08: Multiply 64-bit = 3c latência, 1/c throughput
    — MUL = throughput 1/ciclo no A78
    — Substituir divisão por multiplicação recíproca quando possível:
      n/d ≈ n × (2^32/d) >> 32 (inteiro recíproco)

  REGRA MC-09: Load-Use Hazard = 4c stall
    — ARM64: 4 ciclos entre load e use do valor carregado
    — Solução: intercalar com instruções que não dependem do load

  REGRA MC-10: Loop overhead = compare + branch = 2c/iteração
    — Para loops de 4 iterações: overhead = 50% do trabalho
    — Unroll ×4 reduz overhead para 12.5%

  CODEX DE BENCHMARK (como aplicar):
    1. Medir com mediana 31 amostras
    2. Identificar bottleneck via perf stat
    3. Aplicar a regra MC-XX correspondente
    4. Re-medir: ganho deve ser > 10% para justificar complexidade
    5. Verificar que output é idêntico (correctness invariant)

[#08 METAPHORM — ABSTRAÇÃO COMO METÁFORA DE TRANSFORMAÇÃO]
================================================================================

  METAPHORM: transformar código existente em código mais eficiente
  mantendo a semântica exata mas mudando a "forma" de execução.

  METAPHORM 1: função → macro inline
    ANTES: u32 crc_byte(u32 c, u8 b) { c^=b; for... }
    DEPOIS: #define CRC_BYTE(c,b) do{ (c)^=(b); ... }while(0)
    GANHO: elimina call overhead (push/pop LR, stack frame)
    QUANDO: funções < 10 instruções chamadas em hot path

  METAPHORM 2: if/else → lookup table
    ANTES: if(state==0) f=a; else if(state==1) f=b; ...
    DEPOIS: static const fn_t F[] = {a,b,c,d}; f=F[state];
    GANHO: O(n) comparações → O(1) load
    QUANDO: N ≥ 4 casos e N ≤ 256

  METAPHORM 3: float → Q16
    ANTES: float v = (float)a * (float)b / 65536.0f;
    DEPOIS: q16_t v = Q16_MUL(a, b);
    GANHO: elimina FPU context, sem NaN/Inf handling
    QUANDO: range do valor < 32768 e precisão 1/65536 é suficiente

  METAPHORM 4: recursão → iteração (tail call elimination)
    ANTES: u64 fib(u32 n) { return n<2 ? n : fib(n-1)+fib(n-2); }
    DEPOIS: loop com dois acumuladores
    GANHO: O(2^n) → O(n) | elimina stack overflow
    APLICAÇÃO: fraf_iterate já usa esta forma

  METAPHORM 5: struct de ponteiros → array de struct
    ANTES: {ptr_a, ptr_b, ptr_c}[] — pointer chasing
    DEPOIS: {a,b,c}[] — todos contíguos em memória
    GANHO: 1 cache line contém múltiplos elementos
    QUANDO: acesso sequencial (não random)

  METAPHORM 6: mutex → CAS atômico
    ANTES: pthread_mutex_lock(); ... pthread_mutex_unlock();
    DEPOIS: while(!CAS(&lock, 0, 1)) RAF_WFE(); ... lock=0; RAF_SEV();
    GANHO: elimina kernel involvement para contention leve
    CUIDADO: starvation possível; usar apenas para seções curtas
